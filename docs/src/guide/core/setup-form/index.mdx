---
title: setupForm
---

BlueForm does not expose a single global `<Form />` component. Instead, you create a form engine bounded to a base configuration through a setup step. This is the entry point of any BlueForm-powered form system.

```tsx
import { setupForm, defineMapping } from 'react-headless-form'
import InputField from './InputField'

export const [Form, defineConfig, Section] = setupForm({
  renderRoot: ({ children, onSubmit }) => (
    <form onSubmit={onSubmit}>{children}</form>
  ),
  fieldMapping: defineMapping({
    text: InputField,
  }),
  i18nConfig: {
    t: (message) => message,
  },
})
```

`setupForm` returns a tuple of three values — destructure and name them as you see fit:

| Value | Description |
|---|---|
| [Form](form.mdx) | The form component, bound to your field mapping |
| [defineConfig](define-config.mdx) | Helper for typing nested config fragments |
| [Section](section.mdx) | Component for authoring typed config fragments inside section components |

:::tip
`Form` is the name used throughout this documentation. Since it comes from destructuring the return value of `setupForm()`, you can rename it to anything that fits your codebase.
:::

## Options

| Option | Required | Description |
|---|---|---|
| `renderRoot` | Recommended | How the root form element is rendered |
| `fieldMapping` | Recommended | Maps field type names to React components |
| `i18nConfig` | Optional | Translation behavior for labels and validation messages |

## `renderRoot`

Controls how the root form element is rendered. This keeps BlueForm layout-agnostic and avoids coupling the library to the web platform.

```tsx
renderRoot: ({ children, onSubmit }) => (
  <form onSubmit={onSubmit}>
    {children}
  </form>
)
```

Since `renderRoot` is just a function, it can accept additional props — useful for injecting DevTools, form attributes, or custom wrappers:

```tsx
const formRoot =
  ({ enableDevTool } = {}) =>
  ({ children, onSubmit, formMethods: { control } }) => (
    <>
      <form onSubmit={onSubmit}>{children}</form>
      {enableDevTool && <DevTool control={control} />}
    </>
  )

const [Form] = setupForm({ renderRoot: formRoot() })
```

## `fieldMapping`

Maps field type names to React components. Each entry tells BlueForm which component to use when a field config declares a given `type`.

```tsx
fieldMapping: defineMapping({
  text: InputField,
  select: SelectField,
  datepicker: DatePickerField,
})
```

When a field declares `type: "text"`, BlueForm renders `InputField` for it. The mapping is enforced at compile time — declaring an unknown `type` in form config will produce a TypeScript error.

For how to author field components, see [Field authoring](../field-authoring/).

## `i18nConfig`

Provides translation support for labels, descriptions, and validation messages. Completely optional — if omitted, all text values are treated as plain strings.

```ts
i18nConfig: {
  t: (message, params) => translate(message, params),
}
```

`validationTranslation` maps RHF rule names to translation keys, so validation messages are translated automatically without putting translation keys into field config:

```ts
i18nConfig: {
  t: (key, params) => i18next.t(key, params),
  validationTranslation: {
    required: 'validation.required',
    minLength: 'validation.minLength',
  },
}
```

For a full i18n guide, see [Internationalization](../i18n.mdx).
